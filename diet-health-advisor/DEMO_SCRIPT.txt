[1:00 - ARCHITECTURE]
(show assets/architecture_diagram.png)
Let's walk through the architecture diagram to see how it works under the hood. 
Every request first enters the Security Checkpoint node, which screens for PII, prompt injections, and invalid medical requests.
If a security event is triggered, it routes immediately to the Security Violation Handler to protect the user and system.
If clean, the query routes to our central orchestrator_agent, which manages the overall session.
The orchestrator delegates tasks to two specialized sub-agents: the meal_planner_agent to design the recipes and schedules, and the nutrition_analyst_agent to calculate macronutrients.
Both sub-agents retrieve verified information from our local Model Context Protocol, or MCP server.
Finally, the proposed plan goes to the request_approval node, where the system pauses for human approval. If approved, it moves to the final output node; if revisions are requested, it routes back to the orchestrator.
